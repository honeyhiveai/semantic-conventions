# HoneyHive Semantic Conventions

The canonical `honeyhive.*` attribute names that the HoneyHive
ingestion pipeline accepts. This repo is the single source of truth
for:

- the wire-format attribute names emitted by HoneyHive SDKs / OTLP
- the stability tier (stable vs development) of every attribute
- the public documentation site at
  `https://honeyhiveai.github.io/semantic-conventions/` (deploys on
  push to `main` once HHAI-5108 lands the Pages workflow)

The model files in `model/` are validated and rendered by
[weaver](https://github.com/open-telemetry/weaver) — the
OpenTelemetry-blessed Rust CLI for semantic-convention registries.
Weaver emits markdown into `dist/`, then [Jekyll](https://jekyllrb.com)
wraps each page in the HoneyHive-branded layout under `_layouts/` and
produces the deployable HTML site at `_site/`. This is the same
Jekyll setup GitHub Pages uses — what you see locally is what gets
published.

## Quickstart

```sh
bash install.sh   # downloads the pinned weaver binary into ./bin/
make check        # validates the registry
make build        # renders markdown + builds the HTML site
make serve        # live-preview the HTML site at http://localhost:4000
make stats        # prints attribute counts
make clean        # removes ./dist/ and ./_site/
```

The pinned weaver version is set in `install.sh` (override with
`WEAVER_VERSION=v0.X.Y bash install.sh` if you must). `bin/`, `dist/`,
`_site/`, and `vendor/` are git-ignored — the binary belongs to your
machine, the rendered artifacts are produced fresh by the deploy
workflow.

### Local preview prerequisites

`make build` and `make serve` need a Ruby toolchain to run Jekyll:

```sh
brew install ruby@3.3                              # macOS
export PATH="/opt/homebrew/opt/ruby@3.3/bin:$PATH" # or rbenv / asdf
make bundle                                         # bundle install --path vendor/bundle
make serve                                          # auto-runs `make bundle` if needed
```

`make bundle` is idempotent. The generated `Gemfile.lock` is gitignored
because Jekyll on GitHub Actions resolves freshly against the same
versions pinned in `Gemfile`.

## How generation works

We wire up the same two weaver modes OTel uses for its docs site, then
hand the markdown to Jekyll for branded rendering:

| Target | Command | Output | When to use |
|---|---|---|---|
| `make build-md` | `weaver registry generate markdown dist/` | Full attribute reference under `dist/attributes/` (one page per namespace + a registry index) | Markdown only — useful in CI before the Jekyll step |
| `make build` | `make build-md` then `bundle exec jekyll build` | Branded HTML site under `_site/` | Local builds; what HHAI-5108's deploy workflow uploads to Pages |
| `make serve` | `make build-md` then `bundle exec jekyll serve` | Live preview on `http://localhost:4000/` | Iterating on layout/styles |
| `make update-docs` | `weaver registry update-markdown --target markdown docs/` | In-place snippet replacement inside `docs/*.md` between `<!-- semconv <group_id> -->` and `<!-- END AUTOGENERATED TEXT -->` markers | Curated narrative pages that mix prose with auto-generated tables |

`docs/` is not committed yet (HHAI-5106 reserves the production docs
structure for the next iteration).

## Adding or editing an attribute

1. Pick the right model file in `model/`. Use one file per logical
   bucket (`honeyhive_session.yaml`, `honeyhive_metadata.yaml`, …) so
   git history is readable. Create a new bucket file when you cross a
   bucket boundary; do not split by stability tier.
2. Add the attribute under the existing `attributes:` list. Every
   attribute MUST set:
   - `id` — the wire-format name, including the `honeyhive.` prefix
     and any `bucket.` segment (e.g. `honeyhive.metadata.cost`).
   - `type` — `string`, `int`, `double`, `boolean`, `string[]`, or an
     inline enum (`members:`).
   - `stability` — `stable` or `development`. Match the tier locked
     in honeyhive-ai-docs (PR #273).
   - `brief` — one sentence describing what the attribute represents.
3. Run `make check`. It MUST exit 0.
4. Run `make serve` and confirm the new row reads correctly with the
   stability badge, anchor, examples, and (if enum) the well-known
   values table.
5. Open a PR. CI (HHAI-5104) will re-run `make check` on every push.
   On merge, the deploy workflow (HHAI-5108) republishes the site.

## Layout

```
honeyhiveai/semantic-conventions/
├── install.sh              # idempotent weaver installer (pinned version)
├── Makefile                # check / build / serve / update-docs / stats / clean
├── Gemfile                 # Jekyll + plugins (jekyll-seo-tag, jekyll-sitemap)
├── _config.yml             # Jekyll site config + brand metadata
├── _layouts/
│   └── default.html        # branded HTML wrapper (header, content, footer)
├── _includes/
│   ├── header.html         # logo + nav + GitHub link
│   └── footer.html         # copyright + repo link
├── assets/
│   ├── css/style.scss      # HoneyHive brand colors, typography, sticky stability column
│   └── img/                # logo + favicon (sourced from honeyhive-ai-docs)
├── model/
│   ├── manifest.yaml       # registry name + schema_url
│   └── honeyhive_*.yaml    # one file per logical bucket
├── templates/
│   └── markdown/           # vendored from open-telemetry/semantic-conventions v1.36.0
├── dist/                   # gitignored — weaver markdown output
├── _site/                  # gitignored — Jekyll HTML output (deployable)
└── SPIKE.md                # HHAI-5103 decisions (layout, publish target, weaver pin)
```

`templates/markdown/` was vendored verbatim from
[open-telemetry/semantic-conventions@v1.36.0](https://github.com/open-telemetry/semantic-conventions/tree/v1.36.0/templates/registry/markdown)
by HHAI-5103. HHAI-5106 added Jekyll front-matter blocks
(`layout: default`, `permalink: …`) at the top of the five templates
that produce per-page output — the minimum needed for Jekyll to wrap
each page in the branded layout. The macros and the `snippet.md.j2`
template are unchanged from upstream.

### Branding

Brand colors and assets are pulled from
[`honeyhiveai/honeyhive-ai-docs`](https://github.com/honeyhiveai/honeyhive-ai-docs)
to keep the registry visually consistent with the Mintlify product
docs:

| Token | Source | Value |
|---|---|---|
| primary | `docs.json#colors.primary` | `#b45f06` |
| primary dark | `docs.json#colors.dark` | `#f88707` |
| logo | `logo/dark.png` | copied to `assets/img/honeyhive-logo.png` |
| favicon | `images/favicon.png` | copied to `assets/img/favicon.png` |
| typography | new for this site | `Inter` via Google Fonts, system-font fallback |

Stability badges follow the upstream OTel color language so the visual
vocabulary matches across semconv sites: stable = green, development =
blue, deprecated = red, RC = purple, mixed = amber.

### Source-mapping section (HHAI-5105 hook)

`_layouts/default.html` and `assets/css/style.scss` reserve a
`.source-mapping` callout style for the per-attribute "where this
lands in the product" section that HHAI-5105 will emit from
`annotations.source_mapping`. When that template lands, wrap the
generated section in a `<div class="source-mapping">…</div>` and the
branded styling kicks in for free.
