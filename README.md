# HoneyHive Semantic Conventions

The canonical `honeyhive.*` attribute names that the HoneyHive
ingestion pipeline accepts. This repo is the single source of truth
for:

- the wire-format attribute names emitted by HoneyHive SDKs / OTLP
- the stability tier (`stable` / `release_candidate` / `development`)
  of every attribute, plus an orthogonal `deprecated:` block on
  individual attributes when retired
- the public documentation site at
  `https://honeyhiveai.github.io/semantic-conventions/` (deploys on
  push to `main` once HHAI-5108 lands the Pages workflow)

The model files in `model/` are validated and rendered by
[weaver](https://github.com/open-telemetry/weaver) — the
OpenTelemetry-blessed Rust CLI for semantic-convention registries —
using the same template pipeline that drives
[opentelemetry.io/docs/specs/semconv/](https://opentelemetry.io/docs/specs/semconv/).

## Quickstart

```sh
bash install.sh   # downloads the pinned weaver binary into ./bin/
make check        # validates the registry
make build        # renders the reference site into ./dist/
make serve        # serves ./dist/ on http://localhost:8765/
make stats        # prints attribute counts
make clean        # removes ./dist/
```

The pinned weaver version is set in `install.sh` (override with
`WEAVER_VERSION=v0.X.Y bash install.sh` if you must). `bin/` and `dist/`
are git-ignored — the binary belongs to your machine, the rendered
artifacts are produced fresh by the deploy workflow.

`make serve` runs `python3 -m http.server` against `dist/`, which serves
the raw markdown files. For HTML rendering, point a Mintlify or Hugo
dev server at `dist/`, or use a markdown previewer (`glow`, `grip`,
VS Code's built-in preview, etc.).

## How generation works

We wire up the same two weaver modes OTel uses for its docs site. Both
read from `model/` and share the same vendored templates under
`templates/markdown/`:

| Target | Command | Output | When to use |
|---|---|---|---|
| `make build` | `weaver registry generate markdown dist/` | Full attribute reference under `dist/attributes/` (one page per namespace + a README) | Auto-generated reference pages |
| `make update-docs` | `weaver registry update-markdown --target markdown docs/` | In-place snippet replacement inside `docs/*.md` between `<!-- semconv <group_id> -->` and `<!-- END AUTOGENERATED TEXT -->` markers | Curated narrative pages that mix prose with auto-generated tables |

`docs/` is not committed yet (HHAI-5106 owns the production docs
structure). Create one locally to test snippet injection.

## Adding or editing an attribute

1. Pick the right model file in `model/`. Use one file per logical
   bucket (`honeyhive_session.yaml`, `honeyhive_metadata.yaml`, …) so
   git history is readable. Create a new bucket file when you cross a
   bucket boundary; do not split by stability tier.
2. Add the attribute under the existing `attributes:` list. Every
   attribute MUST set:
   - `id` — the wire-format name, including the `honeyhive.` prefix
     and any `bucket.` segment (e.g. `honeyhive.metadata.cost`).
   - `type` — `string`, `int`, `double`, `boolean`, `string[]`, or an
     inline enum (`members:`).
   - `stability` — see "Stability vocabulary" below.
   - `brief` — one sentence describing what the attribute represents.
3. Run `make check`. It MUST exit 0.
4. Run `make build` and open `dist/attributes/honeyhive.md` —
   confirm the new row reads correctly with stability badge, anchor,
   and examples.
5. Open a PR. CI (HHAI-5104) will re-run `make check` on every push.
   On merge, the deploy workflow (HHAI-5108) republishes the site.

## Stability vocabulary

The registry uses the OpenTelemetry stability vocabulary. Pick exactly
one of:

| Value | Meaning |
|---|---|
| `stable` | Load-bearing. Breaking changes go through a documented migration. Safe to reference in evaluator configuration, dashboards, saved filters, and SDK code. |
| `release_candidate` | Feature-complete, in production use, ironing out edge cases. May still receive minor backwards-compatible adjustments before promotion to `stable`. Renders as an "rc" badge. |
| `development` | Actively designed. Expect breakage; do not bake into long-lived integrations. |
| `experimental` | Alias for `development`, kept for OTel-vocabulary migration tolerance. Prefer `development` for new attributes. |
| `mixed` | **Do not author manually.** Synthesized by the renderer when child attributes of a group disagree. |

`weaver registry check --future` (which `make check` runs) rejects the
legacy `stability: deprecated` form. Use the orthogonal `deprecated:`
block instead — see below.

## Deprecating an attribute

Deprecation is orthogonal to stability: an attribute's `stability:`
keeps documenting how stable the attribute was *while it existed*, and
a sibling `deprecated:` block records that it's retiring. Apply the
block at the attribute level (or, for a whole group, at the group
level):

```yaml
- id: honeyhive.gen_ai.response.usage_cost
  type: double
  stability: stable
  deprecated:
    reason: renamed
    renamed_to: honeyhive.gen_ai.response.cost
    note: Removal scheduled for v2.0 — emitters should switch to the new name.
  brief: Aggregate cost in USD attributed to the response.
```

Recognized fields inside `deprecated:`:

- `reason` (required) — short tag: `renamed`, `obsoleted`, `uncategorized`.
- `renamed_to` (optional) — the attribute that replaces this one.
- `note` (optional) — free-form context (timeline, migration guidance).

The renderer emits a red **Deprecated** badge with the note inline
whenever a `deprecated:` block is present, regardless of the
`stability:` value. **Never combine the legacy `stability: deprecated`
with the new orthogonal block** — pick the block.

## Layout

```
honeyhiveai/semantic-conventions/
├── install.sh              # idempotent weaver installer (pinned version)
├── Makefile                # check / build / serve / update-docs / stats / clean
├── model/
│   ├── manifest.yaml       # registry name + schema_url
│   └── honeyhive_*.yaml    # one file per logical bucket
├── templates/
│   └── markdown/           # vendored from open-telemetry/semantic-conventions v1.36.0
└── SPIKE.md                # HHAI-5103 decisions (layout, publish target, weaver pin)
```

`templates/markdown/` is vendored verbatim from
[open-telemetry/semantic-conventions@v1.36.0](https://github.com/open-telemetry/semantic-conventions/tree/v1.36.0/templates/registry/markdown).
HHAI-5106 will fork these into HoneyHive-specific templates if/when we
need to diverge from the OTel rendering style; until then we benefit
from upstream improvements for free.
